# Phase 03 — Data Model

## Context Links
- [Audit § Tables](research/researcher-current-state-audit.md) (20+ tables, 66 migrations)
- [Audit § Risk: RLS](research/researcher-current-state-audit.md)

## Overview
**Priority:** P1 (highest-risk phase — schema locks all downstream work)
**Status:** pending
Distill 66 migrations into ~5 clean v1 migrations under `thethaomammo` schema. Drop dead fields (Drive IDs, legacy sponsor tier, redundant flags). Keep proven shapes (registrations, matches, courts). Apply RLS per table using Phase 02 helpers.

## Key Insights
- 66 migrations include 6+ RLS rewrites — sign of policy thrash. Greenfield = one shot, get it right.
- Soft-delete via `deleted_at` is universal across audit; keep but enforce via `WHERE deleted_at IS NULL` in every public view.
- `event_payments` table (per-event fees) + `payments` (per-registration) was overlapping. Collapse to single `registration_payments` with FK to registration + event.
- Drive IDs (`proof_drive_url`) — drop entirely. Single Supabase Storage path.
- `team_payment_key` (doubles team bundling) — keep, but as nullable group key.
- Court sponsor tier merged into `courts` — drop standalone field.

## Requirements

### Functional
- Schema `thethaomammo` houses all tournament data.
- Core entities: clubs, athletes, tournaments, events, registrations, teams, matches, match_scores, courts, schedule_breaks, notifications, gallery_photos, sponsors, site_settings.
- All tables have RLS on, deny-by-default.
- Soft-delete (`deleted_at timestamptz`) on athletes, tournaments, clubs.
- `display_id` for athletes (e.g. "CL26050532").
- Public views expose only safe columns to anon.

### Non-Functional
- Migrations idempotent on local re-apply (`drop schema thethaomammo cascade; reapply`).
- Indexes on every FK + common filter (tournament_id, event_id, status, deleted_at IS NULL).
- All FKs `on delete cascade` where safe, `restrict` where not (e.g. athlete → registration).

## Architecture

```
shared.app_grants ──┐
                    │
                    ▼ (helper RLS)
thethaomammo.clubs ──< athletes ──< registrations >── events >── tournaments
                              │              │
                              │              └── registration_payments (proof_url)
                              │
                              ├──< teams (doubles)
                              │
                              └──< match_participants ──> matches ──< match_scores
                                                              │
                                                              └─ court_id → courts
                                                              └─ scheduled_at, status
notifications (user_id, type, payload jsonb)
gallery_photos (tournament_id, storage_path)
sponsors (tournament_id?, tier, logo_url)
site_settings (singleton or kv)
```

Flow:
- Register: Phase 04 RPC writes registrations + (if doubles) teams + registration_payments
- Bracket gen: Phase 05 RPC writes matches + match_participants
- Score: Phase 05 trigger on match_scores → updates match.status, cascades winners/losers

## Implementation Steps
1. Sketch ERD (mermaid in this file or a `visuals/erd.mmd`).
2. Draft `000004_thethaomammo_core.sql`:
   - `create schema thethaomammo;`
   - `thethaomammo.clubs(id, name, slug unique, zalo_phone, logo_path, created_at, deleted_at)`
   - `thethaomammo.athletes(id, display_id unique, full_name, dob date, gender, club_id fk, user_id fk auth.users null, created_at, deleted_at)`
   - `thethaomammo.tournaments(id, slug unique, name, starts_at, ends_at, venue, status, payment_qr_path, zalo_group_url, rules_html, payment_info_text, prize_structure jsonb, created_at, deleted_at)`
   - `thethaomammo.events(id, tournament_id fk, kind enum(singles,doubles,mixed), age_group enum(...), gender enum(male,female,mixed), entry_fee_vnd int, capacity int, created_at)`
3. `000005_thethaomammo_registrations.sql`:
   - `registrations(id, event_id fk, athlete_id fk, status enum(registered,confirmed,withdrew), team_id fk null, payment_status enum(unpaid,pending,paid,rejected), payment_proof_path text, created_at)` + unique `(event_id, athlete_id) where deleted_at is null`
   - `teams(id, event_id fk, name, captain_athlete_id fk, payment_group_key uuid)`
   - `registration_payments(id, registration_id fk, amount_vnd int, paid_at timestamptz, verified_by uuid fk auth.users, note)`
4. `000006_thethaomammo_matches.sql`:
   - `courts(id, tournament_id fk, name, sort_order int, status enum(available,in_use,maintenance))`
   - `matches(id, event_id fk, round int, slot int, court_id fk null, scheduled_at timestamptz, status enum(pending,in_progress,completed), winner_participant_id fk null, third_place bool default false)`
   - `match_participants(match_id fk, slot int, athlete_id fk null, team_id fk null, advanced_from_match_id fk null, primary key (match_id, slot))`
   - `match_scores(id, match_id fk, set_no int, slot1_score int, slot2_score int, recorded_at)`
   - `schedule_breaks(id, tournament_id fk, age_group, starts_at, ends_at, reason)`
5. `000007_thethaomammo_misc.sql`:
   - `notifications(id, user_id fk auth.users, type enum(...), payload jsonb, read_at, created_at)`
   - `gallery_photos(id, tournament_id fk, storage_path, caption, created_at)`
   - `sponsors(id, tournament_id fk null, name, logo_path, link_url, tier enum(gold,silver,bronze,partner), sort_order)`
   - `site_settings(key pk, value jsonb)`
6. `000008_thethaomammo_rls.sql` — for every table:
   - `alter table … enable row level security`
   - Public read for `tournaments`, `events`, `matches`, `match_scores`, `courts`, `gallery_photos`, `sponsors` (where `deleted_at IS NULL`).
   - Admin write via `shared.has_role('thethaomammo','admin')`.
   - Club manager scoped to `shared.user_scope('thethaomammo','club_manager') = club_id`.
   - Athlete reads own `registrations` (via `athletes.user_id = auth.uid()`).
   - Referee writes `match_scores` only.
7. `000009_thethaomammo_views.sql` — `v_tournaments_public`, `v_matches_live`, `v_event_capacity_remaining`.
8. `000010_thethaomammo_realtime.sql` — publish `matches`, `match_scores`, `courts`.
9. Generate types: `pnpm supabase gen types typescript --local > src/lib/types/db.ts`.
10. Mirror zod schemas in `src/lib/types/domain.ts` (shared client/server).
11. Seed `supabase/seed.sql` with 1 tournament + 2 events + 4 athletes for local dev.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Schema gap discovered late (missing column) | H | M | Tight cycle: build Phase 04 features against Phase 03 in parallel after schema lock; add `meta jsonb` escape hatch on tournaments/events |
| RLS recursion via helper | M | H | Phase 02 helpers `SECURITY DEFINER` with `search_path = ''`; do not query tables w/ RLS on inside helpers |
| Drift between zod + db types | M | M | Single source: generate db types, hand-write zod, write a vitest that compares column names |
| Soft-delete leak (forgot `WHERE deleted_at IS NULL`) | H | H | Public views enforce it; ban base-table reads from client in lint rule |
| FK cycle (matches.advanced_from_match_id) | L | M | Cycle-safe insert order; `match_participants.advanced_from_match_id` is nullable |
| Free tier 500 MB cap | L | L | Photos/proof in Storage, not bytea — verified |

## Backwards Compatibility / Migration
Greenfield. [Phase 10](phase-10-data-migration.md) migrates "easy" tables: global config + tournaments/events + per-user athletes/registrations. Hard tables (matches, payments, storage, notifications) explicitly skipped.

### Migration Compatibility
- **Tables migrated** (must keep old column shapes as a superset; new-only columns nullable or default):
  - `tournaments`, `events`, `sponsors`, `age_categories`, `site_settings`, `courts`, `tournament_breaks` — bulk-copied
  - `athletes`, `registrations` — per-user
- **Conditional column:** `tournaments.is_legacy bool default false` — used only if Phase 10 (optional) runs; defaults to `false` for fresh-start. Phase 06/07 read this to hide Live/Print actions on legacy tournaments when present.
- **Athletes superset:** keep `club_name TEXT` column alongside `club_id uuid nullable` — old schema denormalized, new schema normalizes optionally. Phase 10 imports with `club_id=NULL`; users/admin can normalize later.
- **Registrations superset:** `payment_proof_url` nullable; new enum value `unknown` added to `payment_status` for imports where old enum doesn't map cleanly.
- `display_id` format kept compatible ("CL26050532" pattern); trigger has `WHEN (NEW.display_id IS NULL)` guard so import bypasses generation.
- Tables NOT migrated (matches, match_scores, teams, payments, notifications, gallery_photos): design freely, no compat constraints.

## Rollback
`drop schema thethaomammo cascade;` then re-apply prior. Local-only until Phase 06 has data; production rollback acceptable pre-launch.

## Security
- Every base table RLS on; deny-by-default.
- Public views explicitly `security_invoker = true` (PG 15+) so RLS still applies through views.
- `display_id` is non-secret but PII-adjacent — do not expose `dob` in public views.
- Storage buckets created here? No — Phase 04 creates `payment-proofs` bucket; Phase 07 creates `gallery` bucket; Phase 01 created none.

## Unresolved Questions
- Should `display_id` be generated via trigger (deterministic from club + year + seq) or app-side? (Default: trigger, since current code uses CL+YY+MM+seq.)
- Keep `prize_structure` as jsonb or normalize to `prizes` table? (Default: jsonb — read-only display.)
- Do we need `match_history` table separate from `match_scores`? (Default: no — `match_scores` is append-only history.)
- Referee scope: per-tournament or per-event? (Default: per-tournament via `scope_id = tournament_id` in grants.)
- **Migration-relevant column diffs** are confined to `athletes` (the only migrated table); see [Phase 10](phase-10-data-migration.md). Schema simplifications in other tables (drop `proof_drive_url`, collapse `event_payments`+`payments` → `registration_payments`, drop `legacy_sponsor_tier`/`court_sponsor_tier`) are pure greenfield choices — no migration impact.
